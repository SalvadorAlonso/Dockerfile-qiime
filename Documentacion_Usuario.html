    La sección Documentación de Usuario prentende dar las claves necesarias para cualquier usuario que se sienta por primera vez frente a su máquina y sin que tenenga ningún conocimiento sobre cómo descargar o ejecutar el programa Ipython pueda hacerlo de manera sencilla y fácil de seguir.

    DESCARGA E INSTALACIÓN DE LA IMAGEN DOCKER

    Para buscar una imagen por el repositorio de Docker y conocer algunas de sus características como el peso y una breve descripción usamos el comando search con la siguiente estructura, docker search [Nombre_imagen] :

    docker search salvadoralonsomartinez/ipython_qiime

    Para descargar una imagen desde el repositorio de Docker docker pull [Nombre_imagen] :

    docker pull salvadoralonsomartinez/ipython_qiime

    Una vez hayamos descargado la imagen la podemos encontrar en el recopilatorio de imágenes descargadas y así consultar otros de sus atributos como el Tag o el IMAGE ID:

    docker images

    EJECUCIÓN IMAGEN DOCKER

    Para ejecutar una imagen Docker previamente descargada y enlazarla con uno de nuestros puertos debemos emplear la orden docker run de la siguiente manera:

    docker run -p XXXX:8888 [nombre_imagen]

    De nuevo poniendo mi imagen de ejemplo tendrímos:

    docker run -p 8282:8888 salvadoralonsomartinez/ipython_qiime

    TUTORIAL BÁSICO DE QIIME

    A continuación vamos a dar paso a un tutorial básico para familiarizarse con QIIME.

    El tutorial esta compuesto por diferentes partes extraídas de los tutoriales que QIIME pone, de manera gratuita, a disposición del usuario a través de su página web en el apartado Tutorials http://qiime.org/tutorials/

    El propósito de este tutorial no es conocer a fondo la totalidad de las funciones que ofrece el programa, sino establecer un primer contacto y practicar las órdenes más comunes de QIIME mediante el uso de IPython notebook. Para usuarios con experiencia en el uso de esta herramienta se recomienda realizar los tutoriales más avanzados que ofrece QIIME http://qiime.org/index.html.

    Los datos utilizados en el tutorial proceden de Moving Pictures del estudio Human Microbiome, donde se seleccionaban muestras diarias de dos sujetos humanos. Las muestras pertenecían a cuatro sitios diferentes del cuerpo: la lengua, la palma de la mano derecha, la palma de la mano izquierda y el intestino (a través de muestras fecales obtenidas del papel higiénico usado).

    Estos datos fueron secuenciados mediante el protocolo de secuenciación Amplicon descrito en Global patterns of 16S rRNA diversity at a depth of millions of sequences per sample. Una versión más reciente de este protocolo que también funciona con Illumina HiSeq 2000 y MiSeq se puede encontrar en aquí.

    El primer paso es abrir la herramienta notebook del programa IPython, para ello una vez hemos descargado la imagen Docker salvadoralonsomartinez/dockerfile_qiime, abrimos la terminal y la ejecutamos:

    docker run -d -p XXXX:8888 salvadoralonsomartinez/dockerfile_qiime

    Acabamos de asignar un puerto al contenedor generado por la imagen, si todo ha ido correcto nos devuelve una serie de líneas y en una de ellas dice:

    The IPython Notebook is running at: http://0.0.0.0:8888/

    Lo único que nos queda por hacer es abrir un navegador y escribir:

    127.0.0.1:XXXX

    Una vez abierto, si clickeamos sobre la venta de arriba a la derecha 'New Notebook' se abre otra pestaña en la que se nos solicita que introduzcamos información de entrada. Acabamos de crear un fichero al que más tarde le daremos el nombre que deseemos.

    Nuestra imagen Docker contiene los datos que vamos a necesitar para realizar este tutorial, en el caso de que se quisiera trabajar con otros datos, deberían descargarse añadiendo el enlace pertinente.

    Unas herramientas útiles y características de IPython notebook son FileLink y FileLinks, que se usan para ver o descargar información. FileLinks sirve para ver o descargar directorios mientras que FileLink se usa para ver o descargar carpetas individuales. Veamos un ejemplo de cada caso, en primer lugar preparamos las funciones con el comando from. Para ejecutar los comandos que vayamos añadiendo debemos hacer click en la opción "Run Cell":

    from IPython.display import FileLinks, FileLink

    FileLinks('moving_pictures_tutorial-1.9.0')

    FileLink('moving_pictures_tutorial-1.9.0/README.txt')

    En este momento nos desplazaremos desde el directorio moving_pictures_tutorial-1.9.0/ al directorio moving_pictures_tutorial-1.9.0/illumina (mediante el comando chdir que ejecutaremos más adelante), para continuar con los restantes pasos, es decir, hemos accedido a la carpeta illumina que se encontraba en el directorio anterior.

    Necesitaremos preparar las funciones FileLink y FileLinks para que trabajen en este nuevo directorio:

    para ello basta con poner, de una en una, las siguiente lineas en la terminal:

    from functools import partial

    from os import chdir

    Ambos from importan módulos de Python, son necesarios para realizar operaciones específicas. La función partial nos permite crear una función parcial con los parámetros que le indiquemos y asignar esta "función parcial" a una palabra clave que más tarde invocaremos.

    Por otra parte la función chdir cambia el directorio en el que nos encontramos.

    chdir('moving_pictures_tutorial-1.9.0/illumina')

    FileLink = partial(FileLink, url_prefix='moving_pictures_tutorial-1.9.0/illumina/')

    FileLinks = partial(FileLinks, url_prefix='moving_pictures_tutorial-1.9.0/illumina/')

    COMPROBACIÓN DE ERROR EN LOS MAPAS

    En la carpeta que hemos descargado para realizar el tutorial encontramos muestras de metadatos, incluyendo información técnica como cebadores y los códigos de barras asignados a cada muestra, además de las partes del cuerpo de las que se extrajeron las muestras. En este grupo de datos estamos trabajando con muestras microbianas pertenecientes a las cuatro partes diferentes del cuerpo de los dos individuos humanos en distintos momentos. Los metadatos, por tanto, en este caso incluyen un sujeto identificado, un momento en el tiempo, y una parte del cuerpo para cada muestra.

    En el siguiente paso vamos a asegurarnos de que nuestro archivo de mapeo es compatible con QIIME, para ello:

    !validate_mapping_file.py -o vmf-map/ -m map.tsv

    Como vemos, en esta ocasión IPython nos informa de que no hubieron errores, pero si hubieran habido tendríamos que revisar la información, como la dirección HTML, que devuelve la terminal para averiguar los errores encontrados. En ese caso tendríamos que reparar los errores mediante un archivo de texto y reintroducir el archivo de mapeo.

    Para entender mejor de qué estamos hablando realizaremos un ejemplo de un fichero que mostrará errores. Llamaremos validate_mapping_file.py al fichero map-bad.tsv y veremos los errores resultantes:

    !validate_mapping_file.py -o vmf-map-bad/ -m map-bad.tsv

    Como avisamos se han detectado errores, pero IPython no nos indica cuáles son, por lo que tenemos que buscarlos nosotros. Para ello introducimos la ruta HTML:

    FileLinks('vmf-map-bad/')

    Cuando esto ocurra podemos hacer click sobre un fichero que QIIME generará de manera automática llamado "__corrected.txt". En él se encontrarán cambios automáticos realizados por QIIME en algunas línas, esto es debido a que dichos caracteres reemplazados no son identificados por el programa. En nuestro caso el fichero no contendrá errores ya que su fallo es que esta vacío.

    DEMULTIPLEXADO Y FILTROS DE CALIDAD DE SECUENCIAS

    Ahora realizaremos una demultiplexación y filtraremos las secuencias en función de su calidad (por ejemplo haciendo un barcoded emparejando las etiquetas que indican la procedencia con cada una de las muestras).

    Si multiplexar es combinar datos de diferentes fuentes o medios, demultiplexar es lo contrario, es decir, recupera y separa información de una fuente "única" en sus diferentes fuentes de origen.

    En general se obtienen los ficheros fastq separados de las secuencias y de las lecturas de barcode. Nótese que hemos pasado los ficheros mientras estaban comprimidos. split_libraries_fastq.py puede manejar ficheros fastq que estén tanto comprimidos como descomprimidos. La estrategia predeterminada en QIIME para el filtrado de calidad de los datos Illumina se describe en Bokulich et al (2013).

    !split_libraries_fastq.py -o slout/ -i forward_reads.fastq.gz -b barcodes.fastq.gz -m map.tsv

    Normalmente queremos ver los resultados de un comando cuando se ejecuta, para ello realizamos una llamada a la información de salida mediante el comando empleado en ver o descargar directorios (FileLinks).

    Veremos cómo ha separado la información mediante la demultiplexación:

    FileLinks('slout/')

    Y para ver cuántas secuencias hemos cerrado usamos la orden count_seqs.py. :

    !count_seqs.py -i slout/seqs.fna

    OPEN-REFERENCE OTU PICKING

    Tras haber demultiplexado secuencias, continuaremos con un cluster de las secuencias en OTUs (Unidad Taxonómica Operacional).

    Recordemos que el término cluster posee diferentes significados dependiendo del contexto en el que nos encontremos. En genética se refiere a un conjunto de genes que codifican para proteínas similares que colectivamente comparten un función generalizada.

    Existen tres métodos para realizar esta operación en QIIME. Se puede usar De Novo, closed-reference u open-referenced OTU picking. El método que emplearemos será el último que hemos mencionado. Para ampliar conocimientos un artículo que habla sobre los tres métodos se puede encontrar en Rideout et. al (2014).

    Se debe tener en cuenta que el comando que vamos a ejecutar empleará los ficheros seqs.fna que generamos en los pasos anteriores, por lo que si no se completaron correctamente éste no dará los resultados esperados. A su vez especificaremos algunos parámetros para el comando pick_otus.py , que trabaja de manera interna en el programa y se asignó a enable_rev_strand_match el valor booleano "True", lo que permite comparar las secuencias en las dos direcciones (la normal y la inversa), con la información de la base de datos de referencia.

    El siguiente comando puede tarde unos 10 minutos en completarse:

    !pick_open_reference_otus.py -o otus/ -i slout/seqs.fna -p ../uc_fast_params.txt

    El resultado principal que obtenemos de este comando es la tabla OTU, o el número de veces que cada unidad taxonómica operativa (OTU) se observa en cada muestra. QIIME usa el estándar BIOM (Biological Observation Matrix) de representación tablas OTU. Muchas tablas OTU se generan mediante este comando, la que normalmente querremos para trabajar es la siguiente: otus/otu_table_mc2_w_tax_no_pynast_failures.biom. Esta tiene un semifallo de OTUs eliminado.

    El comando open-reference OTU picking también produce un árbol filogenético donde las extremidades son los OTUs. El archivo que contiene el árbol es otus/rep_set.tre y es el fichero que debería usarse con otus/otu_table_mc2_w_tax_no_pynast_failures.biom en los cálculos de diversidad filongenética. El árbol se encuentra en formato newick.

    Para ver la salida que genera este comando, empleamos FileLink sobre el fichero index.html en el directorio de salida:

    FileLink('otus/index.html')

    En este caso no visualizaremos nada si clickeamos sobre el fichero, ya que se trata de un fichero creado a modo de prueba y está vacío.

    Para calcular algunas estadísticas de resumen de la tabla OTU podemos ejecutar el siguiente comando.

    !biom summarize-table -i otus/otu_table_mc2_w_tax_no_pynast_failures.biom

    La parte clave de la información que se necesita extraer de los datos de salida es la profundidad de secuenciación que debe emplearse en la análisis de diversidad. Muchos de los análisis que siguen requieren que haya el mismo número de secuencias en cada muestra, por lo que se necesita revisar las secuencias por muestra y decidir qué profundidad se desea. Cualquier muestra que no posea al menos el mismo número de secuencias no se incluirá en los análisis, por tanto se trata siempre de buscar un equilibrio entre las muestras empleadas y las desechadas.

    A CONTINUACIÓN VAMOS A REALIZAR UN ANÁLISIS DE DIVERSIDAD.

    En esta sección ejecutamos el script core_diversity_analysis.py, que emplea muchos de los "primeros pasos" que los usuarios necesitan para desarrollar los análisis de diversidad. El principal fichero con el que los usuarios interactuarán es con el archivo index.html, que ofrece enlaces a las diferentes partes de los resultados del análisis.

    Téngase en cuenta que en este paso pasamos -e como la profundidad de muestreo que vamos a utilizar para los análisis de diversidad. En este caso se ha seleccionado 1114 como la profundidad en base a la salida que produjo el comando anterior (el de los datos de la tabla-resumen BIOM). Este valor es específico del estudio, por lo que no es obligatorio que empleen el mismo valor en sus análisis.

    Quizás aparezca un cronómetro al introducir el siguiente comando, pero como indica el aviso, no es algo a tener en cuenta. QIIME a veces incluye este tipo de alertas que ayudan a determinar si los análisis son válidos.

    !core_diversity_analyses.py -o cdout/ -i otus/otu_table_mc2_w_tax_no_pynast_failures.biom -m map.tsv -t otus/rep_set.tre -e 1114

    Abrimos el archivo index.html en el directorio resultante, lo que nos llevará a los diferentes resultados:

    FileLink('cdout/index.html')

    Los resultados devueltos tratan todas las muestras de manera independiente, pero en algunas ocasiones es útil categorizar las muestras por su metainformación. Esto puede llevarse a cabo creando categorías (con cabeceras, por ejemplo) a core_diversity_analyses.py con el parámetro -c. Ya que el comando core_diversity_analyses.py tarda bastante en completarse, tenemos una opción de "recuperación del fallo", --recover_from_failure, que puede usarse para re-ejecutar el comando a partir de un punto en el que se produjo un fallo. Esta opción también puede usarse para añadir categorías de análisis si no se incluyeron desde el principio.

    A continuación para probar el comando --recover re-ejecutaremos core_diversity_analyses.py con dos grupos de categorías, uno para "Sampletype" y el otro para DaysSinceExperimentStart.

    !core_diversity_analyses.py -o cdout/ --recover_from_failure -c "SampleType,DaysSinceExperimentStart" -i otus/otu_table_mc2_w_tax_no_pynast_failures.biom -m map.tsv -t otus/rep_set.tre -e 1114

    FileLink('cdout/index.html')

    Es posible que se hayan dado cuenta de que los espacios PCoA generados por core_diversity_analyses.py no coinciden perfectamente con SampleType, basándonos en los que sabemos de microbiología humana, es algo inesperado. Como se trata de series en el tiempo podemos intentar extraer más información colocando un eje temporal en los espacios PCoA y re-ejecutando el análisis sustituyendo los espacios anteriores por la orden Emperor.

    Emperor es una herramienta de visualización de espacios PcoA.

    !make_emperor.py -i cdout/bdiv_even1114/weighted_unifrac_pc.txt -o cdout/bdiv_even1114/weighted_unifrac_emperor_pcoa_plot -m map.tsv --custom_axes DaysSinceExperimentStart

    !make_emperor.py -i cdout/bdiv_even1114/unweighted_unifrac_pc.txt -o cdout/bdiv_even1114/unweighted_unifrac_emperor_pcoa_plot -m map.tsv --custom_axes DaysSinceExperimentStart

    IMPORTANTE: intercambiar los puntos de PCoA, como se sugiere arriba, no es lo mismo que trabajar sin esos puntos.

    Si ha tenido algún problema con la salida generada por uno o varios de los comandos anteriores puede ejecutar la siguiente orden para comprobar si sus datos de salida coinciden con los que deberían salir:

    FileLinks("precomputed-output/")
